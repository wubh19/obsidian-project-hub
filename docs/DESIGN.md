# Project Hub 设计方案

## 1. 项目目标

`Project Hub` 是一个面向 Obsidian Vault 的轻量级项目管理插件。它不引入独立数据库，不要求额外服务端，而是直接复用用户已经在使用的 Markdown、YAML Frontmatter 和 checklist 语法，实现以下目标：

- 让项目、版本、任务、路线图都能基于本地 Markdown 文件管理
- 让项目数据既可被插件消费，也可被用户直接编辑和长期保存
- 在 Obsidian 中提供统一的 Dashboard、Kanban、版本中心和 Roadmap 视图
- 将“展示层能力”和“Markdown 文件事实源”保持一致，避免出现 UI 状态与文件状态脱节

## 2. 设计原则

### 2.1 文件即事实源

所有核心业务数据以 Vault 中的 Markdown 文件为准，插件只做解析、聚合、展示与有限回写。

### 2.2 低侵入约定优先

优先通过目录结构、Frontmatter 字段和 checklist 约定推断业务语义，减少复杂配置。

### 2.3 可读可编辑

即使脱离插件，用户仍然可以直接阅读和修改项目文件，不会被专有格式锁定。

### 2.4 渐进增强

在保留 Markdown 原生工作流的前提下，逐步增加图表、拖拽、快速创建等增强能力。

## 3. 使用场景

### 3.1 项目负责人

- 浏览项目整体健康度和任务完成率
- 查看版本进展和延期任务
- 基于 Roadmap 判断未来版本排期

### 3.2 开发与协作者

- 在版本文件中维护 checklist 任务
- 在看板中拖拽任务状态并回写 Markdown
- 使用快速创建任务减少手工录入成本

### 3.3 运维与交付人员

- 将交付与回归事项直接纳入版本 checklist
- 与研发任务一起进入统一统计和可视化

## 4. 范围定义

当前版本聚焦以下对象：

- `project`：项目主记录
- `version`：版本记录与版本内 checklist 任务
- `roadmap`：路线图表格
- `task`：由 checklist 或历史单任务文件解析出的任务实体

当前版本不包含以下能力：

- 多人实时协同锁机制
- 独立权限体系
- 服务端同步
- 复杂工作流引擎
- 甘特图编辑器和可视化 Roadmap 反向回写

## 5. 信息架构

插件在界面层当前提供 3 个主要模块：

1. 全局统计区（All Projects Summary）
2. 项目 & 版本状态看板（Project Version Board）
3. 任务看板（Task Kanban）

这种布局对应用户的使用顺序：先看全局，再看项目与版本状态，最后进入某个版本处理具体任务。

其中项目 & 版本状态看板采用“项目一行、状态三列”的二维结构：

- 行维度为项目
- 列维度为版本状态（Todo / In Progress / Done）
- 单元格内展示版本卡片，支持按版本号降序排列
- 当某状态下版本卡片超过 3 个时，默认折叠并提供展开按钮
- 点击版本卡片可联动下方版本任务看板，双击可直接打开版本文件

## 6. 数据模型设计

### 6.1 Project

项目实体来源于 `type: project` 文件，也允许通过版本、任务、路线图记录反向补全项目。

关键字段：

- `project`
- `projectPath`
- `owner`
- `status`
- `start`
- `end`

### 6.2 Version

版本实体来源于 `type: version` 文件，既承担版本元数据，也承担该版本 checklist 任务的容器。

关键字段：

- `version`
- `status`
- `start`
- `end`
- `releaseDate`

### 6.3 Task

任务实体统一抽象为 `TaskRecord`，兼容两类来源：

- `version-task`：版本文件中的 checklist
- `task-file`：历史单任务文件

兼容字段：

- `status`
- `owner`
- `priority`
- `due`
- `source`
- `lineNumber`
- `rawText`

### 6.4 Roadmap

路线图实体来源于 `type: roadmap` 文件中的 Markdown 表格。当前只负责解析和展示，不负责结构化回写。

## 7. 目录与约定设计

推荐目录结构如下：

```text
Projects/
  <项目名>/
    00_Project.md
    01_Roadmap.md
    Versions/
      1.0.0.md
    Docs/
      技术方案.md
      会议记录.md
```

设计原因：

- `00_Project.md` 作为项目锚点，便于识别项目主信息
- `01_Roadmap.md` 固定路线图入口，降低查找成本
- `Versions/` 聚合版本与版本任务
- `Docs/` 保留辅助文档，不强制纳入解析对象

当文件位于 `Projects/<项目名>/...` 下时，插件优先按路径推断项目归属；若存在 Frontmatter，则按字段优先、路径兜底。

## 8. 技术架构设计

### 8.1 总体架构

系统采用单插件、单进程、内存聚合的本地架构：

1. Obsidian 插件生命周期启动
2. 扫描 Vault 中所有 Markdown 文件
3. 解析项目、版本、任务、路线图实体
4. 构建内存态 `ProjectStore`
5. 由 Dashboard View 订阅 Store 并完成渲染
6. 用户操作触发文件回写，再由监听器刷新内存态

### 8.2 模块划分

#### 插件入口层

文件：`src/main.ts`

职责：

- 注册视图
- 注册命令和 Ribbon
- 初始化 Store
- 监听 Vault 文件 create / modify / delete 事件

#### 解析层

文件：`src/core/parser.ts`

职责：

- 解析 Frontmatter
- 按路径推断项目归属
- 提取 checklist 任务
- 识别 owner / priority / due / status
- 解析 Roadmap Markdown 表格

#### 数据聚合层

文件：`src/core/store.ts`

职责：

- 保存项目、版本、任务、路线图快照
- 提供统计聚合方法
- 负责 rebuild / refreshFile / removeFile
- 对外提供订阅机制驱动 UI 更新

#### 展示层

文件：`src/views/dashboard-view.ts`

职责：

- 渲染全局统计区
- 渲染项目 & 版本状态二维看板
- 渲染版本任务看板
- 响应项目切换、版本切换、任务拖拽、状态变更等交互

#### 任务创建层

文件：`src/modals/create-task-modal.ts`

职责：

- 收集任务标题、版本、负责人、优先级、截止日期
- 将任务写入目标版本文件
- 在必要时自动创建目录与初始文件

## 9. 核心流程设计

### 9.1 启动与全量构建

1. 插件 `onload`
2. 创建 `ProjectStore`
3. 调用 `rebuild()` 扫描全部 Markdown
4. 解析实体并建立内存索引
5. Dashboard 首次渲染

适用场景：插件启动、手动刷新、数据不一致后的重建。

### 9.2 文件变更增量刷新

1. Obsidian 触发 `create` 或 `modify`
2. Store 先移除旧记录
3. 重新解析目标文件
4. 重建项目聚合结果并触发订阅更新

适用场景：用户在 Obsidian 中直接编辑 Markdown。

### 9.3 快速新建任务

1. 用户选择项目
2. 打开创建任务弹窗
3. 选择目标版本
4. 拼接 checklist 任务行
5. 写回目标 Markdown 文件
6. 自动打开对应文件并刷新 Store

### 9.4 看板拖拽改状态

1. 用户从 Todo/In Progress/Done 中拖拽卡片
2. View 计算目标状态
3. 读取原始文件并定位 checklist 行
4. 修改 `- [ ]` / `- [x]`、`🚧` 与 `✅ YYYY-MM-DD` 标记
5. 保存文件后触发 Store 刷新

该方案保证 UI 交互最终落回 Markdown 文件，而不是只改内存态。

## 10. 解析规则设计

### 10.1 Frontmatter 规则

当前解析器采用轻量实现，支持简单的 `key: value` 结构，不支持复杂嵌套对象和数组。

适用原因：

- 当前业务字段较少
- 可控且易维护
- 不额外引入 YAML 解析依赖

限制：

- 复杂 YAML 兼容性有限
- 多行值和深层嵌套字段不会被完整解析

### 10.2 checklist 规则

当前通过任务文本中的符号约定提取元信息：

- `@张三` 表示负责人
- `🔥` 表示高优先级
- `⚠️` 表示中优先级
- `🚧` 表示进行中（In Progress）
- `📅 YYYY-MM-DD` 表示截止日期
- `✅ YYYY-MM-DD` 表示完成日期

优点：

- 与 Markdown 原生任务格式兼容
- 录入门槛低
- 易于人工编辑

### 10.3 Roadmap 规则

路线图基于 Markdown 表格，至少包含：

- `version`
- `start`
- `end`
- `status`

既支持完整日期，也支持 `MM-DD` 短日期，短日期将按项目起始年份或版本年份补全年份。

## 11. 统计与可视化设计

### 11.1 统计口径

- 完成率 = `done / total`
- 延期任务 = 当前日期之后仍未完成且已设置 `due`
- 版本进度 = 按 `task.version === version.version` 聚合
- 按人统计 = 按 `owner` 维度聚合完成数和总数

### 11.2 燃尽图策略

当前燃尽图使用文件最近修改时间近似表示任务完成时间。

优点：

- 不要求额外记录完成日志
- 可快速形成趋势视图

限制：

- 若任务完成后文件再次被编辑，时间会被覆盖
- 无法精确反映历史完成轨迹

后续可升级为：在任务行中追加完成日期标记，或在 Frontmatter 中引入结构化事件字段。

## 12. 非功能设计

### 12.1 性能

- 初次加载采用全量扫描，适合中小型 Vault
- 日常编辑依赖单文件增量刷新，降低重复解析成本
- 所有聚合均在内存中完成，避免重复遍历 Vault

### 12.2 可靠性

- 所有展示数据都可由 Markdown 重新构建
- 删除文件时同步移除内存记录
- 未声明 `project` 的文件优先按目录推断，降低数据丢失概率

### 12.3 可维护性

- 模块边界清晰，解析、聚合、展示分离
- 类型定义集中在 `src/types.ts`
- 统计逻辑集中在 Store，避免散落到 UI

## 13. 已知限制

- Frontmatter 解析能力较基础
- 任务状态覆盖 `todo`、`in-progress`、`done`
- 看板回写依赖行号，若外部编辑导致任务位置变化，需要重新解析后再操作
- Roadmap 仅支持表格解析，不支持图形化编辑
- 当前未提供插件设置页，目录规则和符号规则尚未可配置

## 14. 演进建议

### 14.1 短期版本

- 增加设置页，允许配置项目根目录和任务标记规则
- 为解析器接入更完整的 YAML 解析能力
- 在版本中心增加筛选、排序与详情跳转
- 为任务创建补充低优先级和紧急优先级的可视展示

### 14.2 中期版本

- 支持自定义任务状态映射，例如 `review`、`waiting`
- 支持多 Roadmap 文件合并展示
- 为任务增加完成时间持久化字段，提升燃尽图准确性
- 增加任务搜索、负责人过滤和逾期过滤

### 14.3 长期版本

- 支持更多项目模板初始化
- 支持跨项目总览视图
- 支持与 Dataview 或其他 Obsidian 生态能力联动
- 支持发布节奏、里程碑和风险项的结构化管理

## 15. 验收标准

设计方案落地后，至少应满足以下验收条件：

1. 插件能自动识别 `Projects/<项目名>/` 下的项目数据
2. 版本 checklist 能被稳定解析为任务
3. 看板拖拽后 Markdown 文件内容正确回写
4. 快速新建任务能写入正确目标文件
5. Dashboard、版本中心、Roadmap 在默认项目数据上可直接展示
6. 删除、修改、创建 Markdown 文件后视图可自动刷新

## 16. 结论

本项目的核心价值不在于构建一个复杂的项目管理系统，而在于用最低的数据迁移成本，把 Obsidian 中已经存在的项目文档、版本文档和 checklist 任务，组织成一套可计算、可展示、可回写的项目管理工作台。

因此，后续设计应始终坚持两个判断标准：

- 新能力是否仍然以 Markdown 为事实源
- 新交互是否能在不破坏用户现有目录和编辑习惯的前提下成立

只要这两个前提不被破坏，`Project Hub` 就能持续保持“文档即系统”的产品定位。
